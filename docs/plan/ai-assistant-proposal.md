> **문서 버전**: v0.2

> **상태**: PR #142 owner 리뷰 1차 반영 — §11 남은 결정 추가 논의 필요

NEXT JUDGE에 학습 보조 AI 도우미를 얹기 위한 기획서. 프로젝트 owner와의
정렬을 위한 **논의 문서**이며, 본구현 진입 전 **§11의 열린 결정들이

제안에서 출발. PR 기반 큐레이션 / 브라우저 채점 / 코드는 서버로 전송이라는
NEXT JUDGE의 기존 아키텍처가 학습용 AI 도우미 도입에 구조적으로 우호적이라는
점이 결정 배경이다.

코드트리가 학습 효과 측면에서 호평받는 패턴이다. 우리는 이 둘을 결합한
포지션을 잡는다.

## 2. 페르소나와 정책 라인

### 페르소나

**소크라테스식 학습 동반자.**

- 정답 코드는 **절대 직접 제공하지 않는다**. 의사코드 골격까지가 한계.
- 사용자가 다음 한 발만 디딜 수 있을 만큼만 밀어준다 — "정답 직행 차단".
- 사용자의 풀이를 부정하지 않는 톤. 회고/대안 제시는 비교가 아니라 추가로.

### 정책 라인

LeetCode 모델을 따라 **모드를 분리**한다.

| 모드 | AI 도우미 | 비고 |
|---|---|---|
| 학습 모드 (기본) | ON | 모든 일반 문제 풀이 |
| 컨테스트 / 평가 모드 (향후) | OFF | URL·세션 차원에서 차단 |

현재는 학습 모드만 존재하므로 정책 라인은 "장차 컨테스트 모드를 도입할 때
구조적으로 분기 가능하게 둔다" 정도로 충분하다. 코드는 이미 서버에 전송되고
있어 모드 플래그 추가는 비용이 낮다.

## 3. 기능 범위 — 두 갈래

### 3-1. 갈래 A: 풀이 중 힌트 ("막혔을 때")

**입력 컨텍스트**

- `problem.md` 본문 + frontmatter
- 사용자가 작성 중인 코드 (에디터 현재 상태)
- 최근 채점 결과 (오답 / 런타임 에러 / 시간 초과 / 컴파일 에러)
- 사용자의 자연어 질문 (있으면)
- (정책) hidden case 실패 시 — **"몇 번째 케이스에서 틀렸다"까지만**.
  입력값 자체는 노출하지 않는다. 노출하면 brute-force 우회 학습이 됨.

**트리거**

- **Pull**: "힌트" 버튼 — 항상 활성
- **Push** (코드트리식): N회 같은 채점 결과 반복 / M분 inactivity 시
  "도움이 필요한가요?" 팝업
  - 사용자별 ON/OFF 토글 필요 (상위권 유저에게는 방해)
  - **MVP 범위 외 — Phase 2 에서 도입** (§12)

**채점 결과별 톤 분기**

| 채점 결과 | 톤 | 비고 |
|---|---|---|
| 컴파일 에러 | 자유롭게 도움 — 알고리즘 누설 위험 0 | |
| 런타임 에러 | 코드 라인 단위 분석 — "여기서 IndexError 가능" | 코드트리 강점 영역 |
| 오답 | 가장 신중. **반례 직접 노출 X.** "이런 종류의 케이스를 생각해봤나요?"로 우회 | |
| 시간 초과 | 복잡도 코칭 — "지금 O(N²)인데 N≤10⁵이면…" | |

### 3-2. 갈래 B: 풀이 후 학습 가이드 ("풀고 나서")

정답을 이미 맞췄으므로 정답 누설 우려가 없어 자유롭다.

- **(a) 내 풀이 회고** — 시·공간 복잡도 진단, 가독성/관용 표현 코멘트.
  "같은 알고리즘 내 개선" 과 "다른 알고리즘으로의 도약" 을 분리해서 제시
  (섞으면 학습자가 길을 잃음).
- **(b) 개념 추상화** — 패턴 이름 부여, 제약이 바뀌면 어떻게 풀이가 변하는지,
  실제 시스템 매핑 (BFS↔라우팅, 해시↔캐시).
- **(c) 다음 추천** — 같은 패턴 점진적 난이도, 협업 필터링(데이터 누적 후),
  사용자 약점 진단.
- **(d) 솔루션 비교** — "본질적으로 다른 접근 N가지". 본인 풀이를 부정하지
  않는 톤.
- **(e) 회고 프롬프트** (옵션) — "다시 푼다면 어디서 시간을 줄였을까요?"
  사용자가 먼저 답하고 AI가 코칭. 학습 효과↑, UX 비용↑ — 강제하지 말고
  토글로.

**1차 범위.** 갈래 A를 먼저 구축하고, 갈래 B는 (a) + (c) 정도로 시작해서
점진 확장한다 (§12 참고).

## 4. 힌트 설계 — Role pool 기반

### 4-1. 왜 고정 N단계가 아닌가

처음 검토했던 "5단계 사다리 (재해석 → 관찰 → 분류 → 자료구조 → 의사코드)" 는
실전에서 두 방향으로 깨진다.

- **쉬운 문제 (Bronze 급)**: 5단계가 의미 없이 같은 말을 반복하게 된다
  (`두 수의 합` 류는 재해석부터 자료구조까지 전부 자명).
- **어려운 문제 (Diamond/Ruby 급)**: 두 번 이상의 통찰 점프가 필요한 문제는
  5칸으로 부족. 관찰(observation)이 여러 개로 갈라지는 경우도 있음.

→ **단계 수를 prompt에 못박지 않고**, role pool에서 LLM이 문제에 맞는 부분
집합을 골라 단계 번호를 매기게 한다.

### 4-2. Role pool

> **결정.** MVP 는 5 개 role 로 시작한다. role 은 string 타입이라 추가 비용이
> 작으므로 좁게 시작해서 운영 데이터를 보고 확장한다.

| Role | 의미 | 적용 예 |
|---|---|---|
| `REFRAME` | 문제 재해석 / 단순화 | "결국 N개 중 K개 골라서 합이 최소가 되도록…" |
| `OBSERVE` | 핵심 관찰 / 성질 (복수 가능) | "정렬하면 어떤 성질이 보일까요?" |
| `CLASSIFY` | 알고리즘 카테고리 | "DP / 그리디 계열" |
| `STRUCTURE` | 자료구조·점화식 형태 + 목표 복잡도 환기 | "dp[i] = …의 최댓값으로 정의. N≤10⁵ 이면 O(N log N) 안에 맞춰야" |
| `SKELETON` | 의사코드 골격 (정답 코드 X) | for / while 구조까지 |

**LLM 은 2~5 개 역할을 골라 정렬한 사다리를 만든다.**

| 난이도 | 예상 사다리 |
|---|---|
| Bronze | `[REFRAME, STRUCTURE]` |
| Gold | `[REFRAME, OBSERVE, CLASSIFY, STRUCTURE]` |
| Diamond+ | `[REFRAME, OBSERVE×2, CLASSIFY, STRUCTURE, SKELETON]` |

### 4-3. 제외한 role 과 후속 처리

| 제외 role | 처리 방식 |
|---|---|
| `REDUCE` (작은 케이스로 축소) | OBSERVE 의 한 표현으로 흡수. 프롬프트가 적절한 경우 "작은 케이스로 줄여 보면…" 톤을 OBSERVE 안에 포함하도록 가이드 |
| `COMPLEXITY` (시간복잡도 환기) | STRUCTURE 에 녹임. 프롬프트가 `time_limit` 신호를 보고 목표 복잡도를 STRUCTURE 단계에서 함께 환기 |
| `EDGE` (경계 케이스 환기) | 알고리즘 통찰이 아니라 구현 디테일. 후속 마일스톤에서 별도 "구현 체크리스트" 영역으로 분리 (§12 Phase 4) |

**프롬프트 설계 함의.** COMPLEXITY 흡수는 결국 STRUCTURE 프롬프트가 복잡도
코칭의 부담을 떠안는다는 뜻이다. 구체적으로 §8-7 프롬프트 spec 에서
"STRUCTURE 단계에서는 `time_limit` 와 `memory_limit` 신호를 함께 보고 목표
복잡도를 환기하라" 는 지시를 박아둔다. §8-6 sourceHash 에 `time_limit` 가
포함된 것과 자연스럽게 맞물린다.

## 5. LLM 그라운딩 — `solution.py` 활용

### 5-1. LLM은 문제를 풀지 않는다

NEXT JUDGE는 **모든 challenge에 검증된 `solution.py`** 가 있다는 점이 다른
저지와 차별화된 자산이다. LLM에게 풀라고 시키는 대신 **정답 풀이를
컨텍스트로 그냥 넘겨준다**.

```
입력 → LLM
  - problem.md
  - solution.py  ← 이미 PR에서 검증된 정답
  - (옵션) contributor가 적은 핵심 관찰 메모
출력 → 힌트 사다리 (role pool 기반)
```

LLM이 하는 일은 "풀이"가 아니라 **"정답으로부터 통찰을 역산해 단계로 풀어
쓰기"**. 후자가 훨씬 안정적이다.

### 5-2. 비교: LeetCode Ask Leet과의 차이

| 항목 | Ask Leet 류 | NEXT JUDGE |
|---|---|---|
| 정답 컨텍스트 | 없음. LLM이 매번 풀이 | `solution.py` 주입 |
| 힌트 사다리 품질 | 모델이 틀리면 통째로 어긋남 | 정답에 그라운딩 |
| 큐레이션 비용 | 사용자 요청마다 매번 추론 | PR 머지 시 1회 |

### 5-3. 함정과 보완책

| 함정 | 영향 | 보완 |
|---|---|---|
| `solution.py`는 단 하나의 풀이 경로 | 힌트가 그 풀이 쪽으로 편향 | (옵션) contributor가 PR에 "다른 가능한 접근들" 메모 |
| 코드에서 "왜"를 역산하는 게 항상 깔끔하진 않음 (세그트리 비트, 파라메트릭 서치 등 고급 기법) | 어려운 문제에서 사다리 품질 저하 | (옵션) contributor의 insight 메모 (자유 텍스트) |
| 모델 업그레이드 시 기존 힌트 품질이 의문 | 사이트 전반 품질 산포 | 모델 버전을 hints.json에 기록, 일괄 재생성 가능 |

## 6. Static vs Dynamic 분리

힌트는 **두 종류로 나뉜다**:

| 종류 | 시점 | 입력 | 비용 | 저장 |
|---|---|---|---|---|
| **Static hint ladder** | PR 머지 시 1회 | problem + solution.py + (옵션 메모) | 문제당 1회 | git 트래킹 + DB |
| **Dynamic feedback** | 사용자 요청 시 | 위 + **사용자 현재 코드 + 채점 결과** | 매 요청 | 캐시 안 함 |

- Static은 "이 문제의 일반적 힌트 사다리". 사용자 무관 동일.
- Dynamic은 "이 사용자가 짠 코드를 보니 L2까지 통찰은 있으니 L3 보여주자",
  "사다리에 없는 다른 접근을 시도 중인데 어디가 어긋났나" 같은 적응형 응답.

**Dynamic은 1차 범위 외**로 둔다 (§12). MVP는 Static + 사용자가 사다리를
자유 열람하는 단순 UX 로 시작 (§7).

## 7. 노출 UX

> **결정.** 자유 열람 + 아코디언 + 좌측 패널 탭 추가. 단계 잠금은 도입하지
> 않는다 — §2 페르소나가 SKELETON 까지로 정답 직행을 이미 막고 있어 UI
> 잠금은 중복 방어선이다.

### 7-1. 노출 위치

좌측 패널에 `힌트` 탭을 추가한다.

```
문제 설명 | 힌트 | 제출 기록
```

URL 쿼리(`tab=hint`) 도 기존 패턴 그대로 확장. "읽는 자료" 카테고리에
일관되게 들어간다.

### 7-2. 노출 방식

자유 열람 + 아코디언 UI:

- 사다리 카드 헤더 (role 라벨) 만 보이고 클릭 시 펼쳐짐
- 어떤 단계가 있는지는 인지하되 강제로 다 펼치지 않아도 됨
- thumbs up/down 피드백을 카드 푸터에 노출 (§9 지표)

### 7-3. push 트리거 / 단계 잠금은 Phase 2

§3-1 의 "N 회 같은 채점 결과 / M 분 inactivity 시 도움 팝업" 은 MVP
범위 외. Phase 2 (UX 다듬기) 에서 도입한다. push 트리거와 단계 잠금은
한 묶음으로 검토 (적극적 개입 동선이 함께 작동).

### 7-4. 모바일

문제 상세 페이지가 `min-w-[425px]` + `overflow-x-auto` 로 데스크톱
레이아웃을 그대로 사용 중이라 좌측 탭에 흡수되면 별도 처리 불필요.

## 8. Static 힌트 파이프라인

### 8-1. 결정: PR 머지 시 자동 생성

> **결정.** Static hint ladder 는 PR 이 main 에 머지되는 시점에 GitHub
> Actions 가 자동 생성한다. lazy generate (첫 요청 시 생성) 는 채택하지
> 않는다. 추가로 PR 단계에서 `validate.yml` 이 `hints.json` preview 를
> PR 브랜치에 자동 커밋해 contributor 가 머지 전에 확인 / 수정할 수 있게
> 한다.

근거:

- 생성 결과를 **사람이 미리 검수**해서 이상한 응답이 사용자에게 노출되는 일을
  막을 수 있음.
- 첫 요청한 사용자가 응답 지연을 겪지 않음.
- 비용이 사용자 트래픽에 비례하지 않고 PR 수에 비례 (예측 가능).

#### 비용 트레이드오프와 절충안 후보

PR 단계 preview + 머지 시 재생성은 **LLM 토큰을 2 회 사용** 하는 구조적
비용을 갖는다 (인지된 트레이드오프). Phase 1 운영에서 실제 PR 빈도와 토큰
단가를 보면서 다음 절충안 중 채택 여부를 별도 PR 로 결정한다.

| 후보 | 내용 | 적합한 상황 |
|---|---|---|
| (1) | Fork PR 만 preview, 자체 브랜치 PR 은 머지 시 1 회만 생성 | push-back 권한 차이를 그대로 활용해 자연스럽게 분리 |
| (2) | PR preview 는 "dry-run" 으로 1~2 개 sample role 만 생성, 머지 시 full ladder 생성 | preview 가치 일부 유지하면서 비용 절감 |
| (3) | PR 단계 preview 자체를 옵트인 (`/generate-hints` 같은 slash command 트리거) | 비용 절감 우선 |

추적은 §11-1.

### 8-2. `hints.json` 파일 패턴

생성된 힌트는 **DB가 아니라 git 트래킹 파일로** 저장하고, DB는 그 거울이다.

```
challenges/<slug>/
  problem.md
  solution.py
  hints.json      ← 새로 추가
  testcases/
```

이 패턴의 장점:

- **리뷰**: PR diff 에서 maintainer 가 hint 내용을 코드처럼 본다.
- **수정**: "이 힌트 이상하다" 싶으면 `hints.json` 한 줄 고치는 PR 로 끝.
- **이력**: git log 로 누가 언제 어떻게 고쳤는지 다 남음.
- **재생성 제어**: `humanEdited: true` 항목은 다음 자동 재생성 시 건너뜀.

### 8-3. 워크플로우

```
[PR 생성: problem.md, solution.py 작성]
        ↓
[validate.yml]
  - 스키마 검증 + 솔루션 실행 (기존)
  - hints.json 이 없거나 sourceHash (§8-6) 가 바뀌었으면
    → LLM 호출 → hints.json 생성/갱신
    → PR 브랜치에 자동 커밋 (bot)
  - Fork PR 은 push-back 권한 없음 → PR comment 로 preview, 머지 후 생성
        ↓
[리뷰: contributor + maintainer 가 hints.json 도 함께 본다]
  - 이상하면 PR 에 직접 커밋해서 수정 (humanEdited: true 마킹)
        ↓
[merge to main]
        ↓
[sync.yml]
  - challenges + hints 모두 DB 로 upsert
```

### 8-4. `hints.json` 스키마

```json
{
  "model": "claude-sonnet-4-6",
  "modelVersion": "2026-01",
  "generatedAt": "2026-05-19T10:00:00Z",
  "sourceHash": "sha256(...)",
  "ladder": [
    {
      "level": 1,
      "role": "REFRAME",
      "content": "...",
      "humanEdited": false
    },
    {
      "level": 2,
      "role": "OBSERVE",
      "content": "...",
      "humanEdited": true
    }
  ]
}
```

| 필드 | 용도 |
|---|---|
| `model`, `modelVersion` | 모델 업그레이드 시 일괄 재생성 대상 식별 |
| `generatedAt` | 생성 시각 — 디버깅용 |
| `sourceHash` | `problem.md` / `solution.py` 변경 감지. 동일하면 재생성 스킵 (§8-6) |
| `humanEdited` | 항목별 — `true` 인 항목은 자동 재생성 시 보존 |

### 8-5. DB 스키마

`challenge_hints` 테이블 신규 (별도 테이블 권장 — `challenges` 본체 비대화
방지):

```
challenge_hints
  id
  challengeId → challenges.id
  level (int)
  role (text — REFRAME | OBSERVE | ...)
  content (text)
  humanEdited (boolean)
  model (text)
  generatedAt (timestamptz)
```

`sync.yml` 이 `challengeId + level` 키로 upsert.

### 8-6. sourceHash 계산 명세

> **결정.** sourceHash 는 sha256 으로 고정. 다음 필드를 정렬·직렬화 후
> 해시한다.

| 필드 | 포함 | 이유 |
|---|---|---|
| `problem.md` 본문 | ✅ | 문제 정의 자체 |
| `solution.py` | ✅ | 정답 그라운딩의 핵심 |
| frontmatter `time_limit` | ✅ | STRUCTURE 단계 복잡도 코칭 톤에 신호. 미래에 채점 enforcement 가 도입되면 sourceHash 정의가 바뀌어야 하므로 미리 포함해 둠 |
| frontmatter `memory_limit` | ✅ | 위와 동일 논리 |
| frontmatter `title` | ❌ | 제목 변경은 문제 정의 변경이 아님 |
| frontmatter `tags` | ❌ | 메타데이터, 힌트 본질과 무관 |
| frontmatter `samples` | ❌ | 예시 입출력은 문제 정의 / 조건 변경이 아님. LLM 입력에는 그대로 포함됨 (별도 결정) |
| `brute.py` / `gen.py` | ❌ | hint 컨텍스트 아님 |

구현 메모:

- frontmatter 직렬화는 키를 알파벳순으로 정렬해 deterministic 하게.
- `problem.md` 본문은 frontmatter 제외 후 원문 그대로.
- role pool 정의 변경이나 모델 변경은 sourceHash 가 잡지 않음 — 일괄
  재생성은 maintainer 수동 트리거로 처리.

### 8-7. 모델 선택과 프롬프트 캐싱

> **결정.** MVP 는 Sonnet 4.6 단일. 난이도 기반 분기는 Phase 2 로 미룬다.

근거:

- 작업이 정답 그라운딩 기반 통찰 역산이라 Opus 는 오버킬.
- Haiku 는 비용 차이가 작은 데 비해 사용자 노출 콘텐츠라 품질 리스크.
- 난이도 분기는 difficulty 필드가 없는 현 상태에서 깔끔하게 정의 불가.
  difficulty 도입 자체가 별도 사안.

#### 프롬프트 캐싱

시스템 프롬프트 + role 정의 + few-shot 예시는 모든 문제에 공통이므로
**Anthropic 프롬프트 캐싱** 으로 cached input 토큰을 90% 할인 받을 수
있다. 프롬프트 prefix 가 deterministic 하도록 설계한다.

캐싱이 적용되는 prefix 구성:

1. 시스템 프롬프트 (페르소나, 정답 직행 금지, role pool 사용 규칙)
2. role pool 정의 (§4-2 / §4-3)
3. few-shot 예시 (있다면)

variable suffix:

4. 해당 문제의 `problem.md` + `solution.py`
5. 출력 포맷 지시 (JSON schema 명세)

#### 운영

- Anthropic API 키는 §8-1 결정대로 main `sync.yml` 에서만 호출되므로
  GitHub Secrets 단순 등록으로 충분.
- 토큰 사용량 / 비용은 Phase 1 운영에서 모니터링 (§8-1 절충안 결정의 입력).

## 10. 미래 옵션: Contributor 직접 작성

### 결정 (현 단계)

> **결정.** MVP 는 LLM-first. contributor 는 `hints.json` 에 직접 작성하지
> **않는다** — 자동 생성된 결과를 검토만 한다. **단**, 향후 contributor 가
> 처음부터 직접 작성할 수 있는 옵션을 열어둘 수 있게 스키마와 워크플로우를
> 호환적으로 둔다.

근거: 서비스 초기에 **contributor 부담을 최소화** 하는 방향이 우선. 힌트
작성을 필수로 두면 PR 진입 장벽이 높아진다.

### 향후 도입 시 고려할 형태

- contributor 가 PR 에 `hints.json` 을 직접 커밋하면 `validate.yml` 은
  LLM 호출을 스킵하고 그대로 통과.
- 또는 일부 항목만 사람이 작성, 나머지는 LLM 이 보강하는 **하이브리드** —
  스키마의 항목별 `humanEdited` 플래그가 이걸 그대로 지원함.
- contributor 가이드 ([CONTRIBUTING.md](../../CONTRIBUTING.md)) 에 "힌트
  직접 작성 시" 섹션 추가.

## 11. 남은 결정 사항 / 논의 필요

> Owner 와 후속 합의 또는 Phase 1 운영 결과 후 결정.

### 11-1. Bot 권한 절충안 채택 여부

§8-1 의 PR preview + 머지 시 재생성은 토큰 2 회 호출 비용을 갖는다 (인지된
트레이드오프). Phase 1 운영에서 실제 PR 빈도와 토큰 단가를 본 후 §8-1 의
절충안 (1) ~ (3) 중 채택 여부를 별도 PR 로 결정.

### 11-2. Dynamic feedback 범위

§6 에서 1 차 범위 외로 두기로 했지만, "사용자 코드를 보고 어디서 막혔는지
짚어주기" 는 결국 차별화 핵심. 도입 시점·UX 를 별도 후속 문서로 다룬다.
**§11-3 (사용자 코드 LLM 전송 정책) 이 사전 조건**.

### 11-3. 사용자 코드 LLM 전송 정책 (신설)

Dynamic feedback 과 학습 모드 본격 도입의 공통 사전 조건. 다음 하위 항목
합의 필요:

- **사용자 동의 UX** — one-time / per-session / persistent toggle 중 어떤
  단위로 동의를 받을 것인가
- **데이터 보존 정책** — LLM 요청·응답 로그 보관 기간, 익명화 여부
- **옵트아웃 메커니즘과 기본값** — opt-in 인가 opt-out 인가
- **개인정보보호법 (PIPA) 적합성 점검** — 알고리즘 코드는 PII 우려가 낮지만,
  사용자가 코드 주석에 개인정보를 적는 케이스 등 고려
- **컨테스트 / 평가 모드 OFF 정책과의 정합** (§2 정책 라인)

[CLAUDE.md](../../CLAUDE.md) 가 이미 "코드 자체는 서버에 전송됨 — AI 학습
모드 지원 예정" 이라고 명시하고 있어, 이 정책 합의가 학습 모드 본격화의
게이트가 된다.

### 11-4. 이벤트 트래킹 인프라 도구 선정 (신설)

§9 지표 수집을 위한 인프라. 후보:

| 도구 | 장점 | 단점 |
|---|---|---|
| Vercel Analytics | 설치 trivial, 기존 배포와 같은 platform | custom event 폭 제한적, Pro tier 필요 |
| PostHog | open source, free tier 넉넉, 이벤트·funnel·session replay 풍부, self-host 옵션 | 도입 / 운영 학습 비용 |
| DB 로깅 + 자체 분석 | 전적 통제 | 대시보드 자체 작성 부담 |

NEXT JUDGE 의 OSS 정체성과 §9 funnel 분석 요구를 고려하면 PostHog 가 우선
후보. 별도 PR 에서 도구 선정 + 도입 PoC.

**Phase 1 exit criteria 에 "이벤트 계측 인프라 가동" 포함** (§12).
